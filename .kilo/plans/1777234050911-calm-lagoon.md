# Plan: MCP-Driven Testing + Model Cost Optimization + Remaining Simplifications

## Phase 0: Model Cost Optimization (CRITICAL — implement first)

### Context

The project's `kilo.jsonc` sets the default model to `deepseek/deepseek-v4-pro`. This means the "code" agent (the default agent used for most sessions) runs on the expensive Pro model for ALL operations — reading files, making edits, running commands, and delegating tasks. The cheaper `deepseek-v4-flash` is already configured for subagents (worker, explore, compaction) but the orchestrator itself burns Pro credits for trivial operations.

The project already has the correct agent architecture:
- `plan` (primary, Pro) — system design, architecture
- `reviewer` (subagent, Pro) — code review
- `worker` (subagent, Flash) — implementation
- `explore` (subagent, Flash) — codebase exploration
- `ask` (primary, Flash) — read-only research
- `compaction` (Flash) — context summarization

### 0-1: Swap Default Model to Flash

**File:** `kilo.jsonc`

Change:
```jsonc
"model": "deepseek/deepseek-v4-pro",  // current
```
To:
```jsonc
"model": "deepseek/deepseek-v4-flash",  // new
```

**Impact:** The "code" agent (default) and any agent without an explicit model override will use Flash instead of Pro. Since all agents in the `agent` block already have explicit models, this only affects the default "code" agent.

### 0-2: Verify Agent Model Assignments

The `kilo.jsonc` `agent` block already correctly assigns:

"agent": {

    "ask":        { "model": "deepseek/deepseek-v4-flash" },   // stays Flash

### 0-5: Worker Delegation Pattern

The workflow rules (`.kilo/rules/10-workflow.md`) already prescribe:
> "Delegate each plan step to the worker agent via Task tool."

When running as the code agent on Flash:
1. Read the plan file
2. Create a todo list of steps
3. For each step: `Task(subagent_type="worker", prompt="...")` — worker runs on Flash
4. Aggregate results
5. Run `/review` → reviewer agent (Pro) for code review

The code agent only handles plan parsing, task delegation, and result aggregation — all low-complexity operations well within Flash capabilities.

### 0-6: Escalation Path for Complex Tasks

If a task is too complex for Flash to handle:
1. Worker fails twice (per escalation policy in `10-workflow.md`)
2. Code agent escalates: splits the task into smaller pieces, or re-routes via `/plan` for a stronger model's design input
3. For genuinely hard single-file changes, the code agent (Flash) can attempt them directly — simple edits are well within Flash capabilities

## Phase A: MCP-Driven Test Corpus Generation

Use the GitHub MCP tools to build a diverse, representative test corpus. Each repo in the corpus should have known ground truth across all dimensions our tools produce.

### A1: Corpus Selection

Use `github_search_repositories` to find repos covering these categories:

| Category | Query | Why |
|----------|-------|-----|
| High-star Python | `language:python stars:>5000` | Validates ranking places well-known repos at top |
| Archived | `archived:true` | Validates archive detection in activity scoring |
| No license | (manual check from search results) | Validates license scoring warns appropriately |
| TypeScript/React | `language:typescript topic:react stars:>1000` | Validates framework detection |
| Go | `language:go stars:>500` | Validates multi-language pattern extraction |
| Rust | `language:rust stars:>500` | Validates framework detection for compiled languages |
| Small/new | `pushed:>=2026-01-01 language:python stars:<50` | Edge case: low-signal repos |
| Well-structured | Manual: `tiangolo/fastapi`, `psf/requests`, `encode/httpx` | Known good references |

**Target: 10-12 repos** representing the full diversity of inputs.

### A2: Ground Truth Collection

For each corpus repo, collect via GitHub MCP tools:

1. **Metadata** (`github_get_file_contents` on repo root): stars, forks, open issues, license SPDX, archived status, pushed_at
2. **README** (`github_get_file_contents` on README.md): full text for section parsing validation
3. **Top-level files** (`github_get_file_contents` on root directory): directory listing for structure validation
4. **Framework markers** (search for config files): package.json, pyproject.toml, Cargo.toml, go.mod, etc.
5. **Commit recency** (`github_list_commits`): most recent commit date for activity validation

Store this as a JSON fixture at `tests/corpus/ground_truth.json`.

## Phase B: Systematic Testing

### B1: Metadata Accuracy Tests

Run `repo_inspector.inspect_repo()` against every corpus repo and assert:

| Assertion | Tolerance |
|-----------|-----------|
| `result.stars == ground_truth.stars` | Exact |
| `result.forks == ground_truth.forks` | Exact |
| `result.license_name == ground_truth.license` | Exact SPDX match |
| `result.archived == ground_truth.archived` | Exact |
| `result.language == ground_truth.language` | Exact (case-insensitive) |
| `result.verdict in ("useful", "maybe", "skip")` | Always present |

### B2: Activity Scoring Calibration

For each repo, compare `_evaluate_activity(pushed_at)` output against actual commit recency from `github_list_commits`. Check that:
- Repos pushed within 30 days → "active"
- Repos pushed within 90 days → NOT "stale"
- Repos pushed > 365 days ago → "stale" or "moderate"

### B3: Framework Detection Accuracy

Run `detect_framework()` on cloned repos (where possible) and compare against manually-checked ground truth from the corpus.

### B4: README Section Extraction

For repos with well-known READMEs (e.g., FastAPI, Requests), verify that `parse_readme_sections()` extracts at least the sections listed in the ground truth.

### B5: Search Ranking Sanity

Run `find_repos_for_task` with 5 diverse queries (e.g., "Python HTTP client", "React component library", "Rust CLI tool") and verify:
- Returns results (non-empty for reasonable queries)
- Top result has `verdict` of "useful" or "maybe"
- Results include the expected repo if it should rank highly (e.g., `encode/httpx` for "Python HTTP client")
- No result has `verdict` of "skip" in top 5 for reasonable queries

### B6: Deep Inspect Coverage

Run `deep_inspect_repo` on at least 2 corpus repos and verify:
- `tree_visual` is non-empty string
- `framework` is detected correctly
- `full_file_snippets` contains at least 1 entry
- Cleanup happens (no temp dirs left behind)

**New test file:** `tests/test_mcp_driven.py` — parametrized across corpus repos, marked `@pytest.mark.integration`

## Phase C: Fine-Tuning Based on Test Outcomes

### C1: Ranker Weight Calibration

If integration tests reveal systematic issues (e.g., archived repos still get "useful" verdicts), adjust:
- The archive penalty in `score_activity()` (currently returns 0.0, which is correct)
- The relevance scoring weights if search results don't match expectations
- The verdict thresholds if too many repos get "useful" vs "skip"

### C2: Framework Detection Gaps

If `detect_framework()` misses frameworks in the corpus:
- Add missing markers to `_FRAMEWORK_MARKERS`
- Fix false positives (e.g., a repo with go.mod + pyproject.toml should not return "go") — this is already handled but verify

### C3: Pattern Extraction Quality

If `extract_patterns_from_repo()` returns empty or "skip" for repos known to be well-structured:
- Tune `verdict = "useful" if len(patterns) >= 2` threshold (currently too strict for repos with detailed but single-section READMEs?)

## Phase D: Remaining Structural Simplifications

### D1: Split `pattern_extractor.py` into focused modules

**Current:** 436 lines mixing 6 concerns.

**Target:**
```
src/repo_finder/
    pattern_extractor.py     (~150 lines) — extract_patterns, distill_patterns, README parsing
    framework_detector.py    (~120 lines) — detect_framework, _FRAMEWORK_MARKERS, _collect_all_files, _read_file
    language_patterns.py     (~80 lines)  — _identify_language_patterns
```

`extract_patterns_deep()` stays in `pattern_extractor.py` and imports from the new modules.

### D2: Extract shared URL utilities

**Current:** `cloner._sanitize_url` and `repo_inspector._parse_owner_repo` do overlapping URL parsing with different return types.

**Target:** New `src/repo_finder/urls.py`:
```python
def normalize_url(url: str) -> str  # strip .git, https://, trailing /
def parse_owner_repo(url_or_slug: str) -> tuple[str, str] | None  # extract owner/repo
```

Migrate both from their current homes. Update all call sites.

### D3: Create dedicated constants module

**Current:** `__init__.py` holds `SKIP_DIRS` and `_now_iso()` alongside its docstring.

**Target:** New `src/repo_finder/constants.py`:
```python
SKIP_DIRS: set[str] = {...}
def now_iso() -> str
```

Remove from `__init__.py`. Update all imports. The `_` prefix on `_now_iso` can be dropped since it's in a constants module.

### D4: Simplify `github_client.py` singleton

**Current:** 18 lines of global state management (`_client`, `_loop_id`, event-loop-aware recreation).

**Target:** Use FastMCP's lifespan hook if available, or simplify to a single module-level `GitHubClient` instance. The event-loop detection was needed for Windows test environments but the project now uses `pytest-asyncio` which manages a single event loop. The complexity can be reduced to:

```python
_client: GitHubClient | None = None

def get_client() -> GitHubClient:
    global _client
    if _client is None:
        _client = GitHubClient()
    return _client
```

If the Windows event loop issue still manifests, add a context-manager-based lifespan in `__main__.py` that creates/closes the client per server run.

### D5: Add cache support to remaining tools

`compare_github_repos`, `extract_patterns_from_repo`, and `deep_inspect_repo` all call `repo_inspector.inspect_repo()` or `pattern_extractor.extract_patterns()` — which already use cache internally where applicable. The tools themselves just don't add a top-level cache key. This is acceptable (the slow operations are cached at the module level) — no change needed beyond documenting this.

## Phase E: Continuous Improvement Loop

### E1: Test Automation Script

Create `scripts/run_quality_checks.py`:
1. Load corpus from `tests/corpus/ground_truth.json`
2. Run `inspect_repo` on all corpus repos
3. Report pass/fail for each metadata assertion
4. Output JSON report to `reports/quality_check_<date>.json`

### E2: GitHub Actions Workflow

Create `.github/workflows/quality.yml`:
- Runs on schedule (weekly) and on PR to main
- Executes `scripts/run_quality_checks.py`
- Requires `GITHUB_TOKEN` as secret
- Posts summary as PR comment or workflow summary

### E3: Regression Bucket

Add a `tests/corpus/regression_bucket.json` with 5 "must-not-regress" repos. These are repos where the verdict or key metrics changed between versions. The quality check script compares current output against stored expected output and flags any change > threshold.

### E4: Iterative Fine-Tuning Loop

```
1. Weekly quality check runs → generates report
2. Review report for regressions or new failures
3. If ranker mis-scores: adjust weights via env vars (no code change)
4. If framework detection misses: update FRAMEWORK_MARKERS
5. If pattern extraction weak: tune thresholds
6. Commit fixes → quality check validates
```
